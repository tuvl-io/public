Core Capabilities

Built for developers.
Designed for production.

YAML → Live API

Define a workflow in YAML. Run tuvl dev. Your HTTP endpoint is instantly live — no controllers, no boilerplate, no serialization code.

Featherlight Execution

No heavy message brokers or bloated background workers. tuvl executes your workflows with pure, signal-driven efficiency on top of a lightning-fast FastAPI core.

Total Auditability

AI shouldn't be a black box. With deep state snapshotting and OpenTelemetry-ready design, every routing decision, latency spike, and context mutation is fully auditable.

Cryptographic Security

Hardened by design. Whether executing a REST call or streaming a deep workflow trace, every interaction is secured and scoped natively via Ed25519 Biscuit tokens.

Dynamic Data Models

YAML-defined schemas auto-generate PostgreSQL tables and full CRUD REST APIs at startup. One definition drives the whole stack — no migrations to write.

Local-First, Any LLM

Choose any model — tuvl uses LiteLLM as its model layer, so a single model: string unlocks 100+ providers: ollama/llama3, gpt-4o, claude-3-5-sonnet, gemini/gemini-2.0-flash, groq/llama3-70b-8192 and more. Your data never leaves your network unless you choose.

MCP & Tool Integrations

Model Context Protocol support built in. Connect LLMs to any external tool or API as a first-class workflow step — no glue code needed.

Developer UI — tuvl insight

tuvl dev serves a built-in developer dashboard at /ui/. Protected by a one-time security key printed on startup. Inspect live step events, browse workflows and models — no external tooling needed.

Human-in-the-Loop

Pause any workflow at a designated step and await a human decision. The dev UI surfaces a live review queue — approve, reject, or supply context, then the engine resumes exactly where it paused. Timeouts and Redis-backed state included.

LLM-as-a-Judge Testing

Write YAML test cases with stub data — no real LLM, database, or HTTP calls needed. Run tuvl test to replay the trace and let a judge model verify each step's output against natural-language assertions.

Production Observability

Two correlated pillars: structured JSON logs (structlog) and distributed traces (OpenTelemetry). Every log line carries trace_id + span_id. Workflow executions produce a workflow.execute parent span with node.<kind> children — context snapshots included. LiteLLM's gen_ai.* spans surface per-model token usage in the same trace. PII fields marked secure: true are masked before any data reaches the collector.

# workflows/contact_intake.yaml
kind: "Workflow"
metadata:
  name: "contact_intake"
context: "Contact"
trigger:
  path: "/api/contacts"
  method: "POST"

steps:
  - id: "validate"
    kind: "functional"
    runner: "validate_contact"
    routes:
      valid:   "save"
      invalid: "END"

  - id: "save"
    kind: "functional"
    runner: "save_contact"
    routes:
      default: "notify"

  - id: "notify"
    kind: "functional"
    runner: "send_welcome_email"
# nodes/contact_nodes.py
from tuvl_engine.nodes.base import node

@node("validate_contact")
async def validate_contact(ctx: dict) -> str:
    """Return signal based on validation result."""
    if not ctx.get("email") or "@" not in ctx["email"]:
        return "invalid"
    return "valid"

@node("save_contact")
async def save_contact(ctx: dict) -> dict:
    repo = ctx["_repo"]("Contact")
    contact = await repo.add({
        "email": ctx["email"],
        "name":  ctx["name"],
    })
    ctx["id"] = str(contact.id)
    return ctx

@node("send_welcome_email")
async def send_welcome_email(ctx: dict) -> dict:
    ctx["notified"] = True
    return ctx

# workflows/screening.yaml
kind: "Workflow"
metadata:
  name: "screen_candidate"
context: "Candidate"
trigger:
  path: "/api/candidates/screen"
  method: "POST"

steps:
  - id: "save_draft"
    kind: "functional"
    runner: "save_candidate"
    routes:
      default: "ai_vetting"

  - id: "ai_vetting"
    kind: "agent"
    agent:
      # any LiteLLM model string — e.g. gpt-4o, claude-3-5-sonnet,
      # gemini/gemini-2.0-flash, groq/llama3-70b-8192, ollama/llama3
      model: "ollama/llama3"
      prompt: |
        Evaluate {{ full_name }} ({{ experience_years }} yrs).
        Classify as senior | needs_review | unqualified.
        Return JSON: {"decision": "..."}
      output:
        format: json
        signal_from: decision
      retry: {attempts: 3, on: [parse_error]}
      context_injection: [resume_text]  # RAG grounding
    routes:
      senior:       "fast_track"
      needs_review: "manual_review"
      unqualified:  "END"
# nodes/screening.py
from tuvl_engine.nodes.base import node

@node("save_candidate")
async def save_candidate(ctx: dict) -> dict:
    repo = ctx["_repo"]("Candidate")
    record = await repo.add({
        "email":            ctx["email"],
        "full_name":        ctx["full_name"],
        "experience_years": ctx.get("experience_years", 0),
        "status":           "pending",
    })
    ctx["id"] = str(record.id)
    return ctx

@node("fast_track_candidate")
async def fast_track_candidate(ctx: dict) -> dict:
    repo = ctx["_repo"]("Candidate")
    await repo.update(ctx["id"], {"status": "fast_tracked"})
    ctx["status"] = "fast_tracked"
    return ctx
TypeScript SDK

@tuvl/client

Full auth lifecycle, workflow execution, and real-time step streaming — all from one npm package. No Biscuit parsing, no raw fetch calls, no wiring.

  • ✓TuvlAuth — login, OAuth2 PKCE, token refresh, logout, getMe() — Biscuit parsing handled server-side, zero WASM in your app
  • ✓TuvlClient — execute any workflow with a single execute() call; REST, SSE, or gRPC-Web negotiated automatically
  • ✓Zero raw fetch / zero token plumbing — the SDK owns the full token lifecycle including refresh and revocation
  • ✓Live step streaming — onProgress fires for every engine event with step id, signal, and duration
  • ✓Full TypeScript types — StepEvent, MeResponse, WorkflowResult, WorkflowManifest and more
SDK Quickstart → API Reference →
app.ts 
// npm install @tuvl/client
import { TuvlAuth, TuvlClient } from "@tuvl/client";

const auth = new TuvlAuth({ baseUrl: "http://localhost:8000" });

// Password login — returns a Biscuit bearer token
const { access_token } = await auth.loginWithPassword(
  "me@example.com", "secret"
);

// Decode roles + scopes without touching Biscuit internals
const me = await auth.getMe(access_token);
// { user_id, groups: ["hr_manager"], scopes: ["candidate:read"] }

// Execute a workflow — streams live step events via SSE
const client = new TuvlClient({ baseUrl: "http://localhost:8000", token: access_token });
const result = await client.execute("screen-candidate", {
  payload:    { candidate_id: 42 },
  onProgress: (ev) => console.log(`[${ev.step_id}] ${ev.signal} (${ev.duration_ms}ms)`),
});

# models/contact.yaml
kind: "ModelDefinition"
version: "v1"
metadata:
  name: "Contact"
spec:
  tablename: "contacts"
  fields:
    - name: "id"
      type: "uuid"
      primary_key: true
      default: "uuid4"
    - name: "email"
      type: "string"
      unique: true
      required: true
    - name: "name"
      type: "string"
      required: true
    - name: "priority"
      type: "string"
      input: false
      description: "AI-assigned priority level"
# workflows/contact_intake.yaml
kind: "Workflow"
context: "Contact"
trigger:
  path: "/api/contacts"
  method: "POST"

steps:
  - id: "save"
    kind: "functional"
    runner: "save_contact"

  - id: "prioritize"
    kind: "agent"
    agent:
      # any LiteLLM model string — e.g. gpt-4o, claude-3-5-sonnet,
      # gemini/gemini-2.0-flash, groq/llama3-70b-8192, ollama/llama3
      model: "ollama/llama3"
      prompt: |
        Contact: {{ name }} — {{ email }}
        Assign a priority: high | medium | low
        Return JSON: {"priority": "..."}
      output:
        format: json
        map:
          priority: priority
    routes:
      default: "enrich"

  - id: "enrich"
    kind: "functional"
    runner: "enrich_contact"
